Benchmarks

The library is designed to be fast. While it is unlikely that the time spend in the histogram is limiting an application, we cannot predict how the library is used.

The following plot shows results of a benchmark on a 9 GHz Macbook Pro. Random numbers from a uniform and a normal distribution are filled into histograms with 1, 3, and 6 axes. 100 bins are used in the first case, 100x100x100=10^6 in the second, and 10x10x10x10x10x10=10^6 in the third. Each test is repeated 50 times and the minimum time is taken.

Plot legend:

root

ROOT classes (TH1I for 1D, TH3I for 3D and THnI for 6D)

py:numpy

numpy functions (numpy.histogram for 1D, numpy.histogramdd for 2D, 3D, and 6D)

py:hd_sd

boost::histogram::dynamic_histogram with boost::histogram::adaptive_storage, called from Python

hs_ss

boost::histogram::static_histogram with boost::histogram::array_storage<int>

hs_sd

boost::histogram::static_histogram with boost::histogram::adaptive_storage

hd_ss

boost::histogram::dynamic_histogram with boost::histogram::array_storage<int>

hd_sd

boost::histogram::dynamic_histogram with boost::histogram::adaptive_storage

boost::histogram::static_histogram is always faster than boost::histogram::dynamic_histogram and safer to use, as more checks are done at compile time. It is recommended when working in C++ only. boost::histogram::adaptive_storage is faster than boost::histogram::array_storage for histograms with many bins, because it uses the cache more effectively due to its smaller memory consumption per bin. If the number of bins is small, it is slower because of overhead of handling memory in a dynamic way.

The histograms in this library are mostly faster than the competition, in some cases by a factor of 2. Simultaneously they are more flexible, since binning strategies can be customised. The Python-wrapped histogram is slower than numpy's own specialized function for 1D, but beats numpy's multi-dimensional histogramming function by a factor 2 to 3.
